# 实现执行清单

## 当前目标

持续推进，直到把现有工程规划中定义的模块全部落成到仓库里，并且每一阶段都做最小原子检查。

## 执行顺序

1. 文档与仓库入口整理
2. `common / datasets / frontends / mapping / primitives`
3. `graph1_evidence`
4. `graph2_match`
5. `memory / supervision / losses`
6. `trainers / inference / tools`
7. `tests / 原子检查 / 文档回填`

## 每阶段原子检查

### 1. 文档与仓库入口整理

- 原始长文档已归档到 `docs/归档/`
- `README.md` 与 `docs/README.md` 链接有效
- `docs/` 分类清晰且无重复职责

### 2. 基础数据与几何模块

- 核心 dataclass 可导入
- dataset / cache / map / primitive 接口可实例化
- 基础几何函数可在单元测试中跑通

### 3. Layer-1

- 可从 `frame + masks + primitives + tracks` 构建 evidence graph
- 可输出 signed primitive graph
- 可得到 primitive cluster ids
- 可构建 `CurrentObjectHypothesis`

### 4. Layer-2

- 可构建 association graph
- 可完成 candidate pruning
- 可分解 connected components
- 可在小组件上给出 graph matching 结果
- 可在大组件或简单组件上回退 Hungarian

### 5. Memory / Supervision / Losses

- track bank 能完成 `query / update / lifecycle`
- teacher replay 与标签构建能输出结构化监督信号
- stage-1 / stage-2 loss 可单独调用

### 6. Trainers / Inference / Tools

- `OnlineEngine.step()` 能贯通主流程
- 训练入口至少能跑通一个 dry-run batch
- 调试工具能导出 debug packet

### 7. 测试与回填

- `python3 -m compileall src` 通过
- 单元测试通过
- 文档更新到与当前实现一致

## 实施约束

- 每完成一个大模块，就先做语法检查和最小行为检查，再进入下一个模块。
- 优先保证接口闭环和可调试性，再逐步增强算法细节。
- 与文档存在偏差时，以“先在文档中记录清楚，再改代码”为准。
- 本地不做环境配置；依赖安装和运行检查默认留给远程服务器或工作站执行。
